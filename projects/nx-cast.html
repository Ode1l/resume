Overview项目概述

NX-Cast is an open-source media receiver for Nintendo Switch homebrew, built around a generic DLNA DMR foundation. The project aims to make playback control and runtime state stay aligned, so control points see an accurate, protocol-compliant device state while the renderer runs real media playback through libmpv. NX-Cast 是一个面向 Nintendo Switch Homebrew 的开源媒体接收器，核心是通用化的 DLNA DMR 基础能力。项目目标是让控制协议状态与真实播放运行时保持一致，使控制端看到准确、符合协议的设备状态，同时由 libmpv 驱动实际媒体播放。

What I Built我完成的核心能力

  • Implemented SSDP discovery, runtime-served `Description.xml`, service control descriptions, SOAP actions, and GENA subscriptions for DLNA DMR control flow.实现 SSDP 发现、运行时 `Description.xml`、服务控制描述、SOAP 动作与 GENA 订阅，完成 DLNA DMR 控制链路。
  • Built a direct `URL -> renderer -> libmpv` playback path, using libmpv as the playback, demux, and decode engine.构建 `URL -> renderer -> libmpv` 的直接播放路径，使用 libmpv 作为播放、解复用与解码核心。
  • Designed a renderer snapshot and event bridge so protocol-observed state reflects runtime playback properties such as duration, pause, mute, seekability, and end-of-file transitions.设计 renderer 快照与事件桥接，使协议可观察状态能够反映时长、暂停、静音、可 seek 性和播放结束等运行时属性。
  • Integrated OpenGL/libmpv render API on Switch while keeping the project compatible with the current devkitPro/libnx toolchain.在 Switch 平台集成 OpenGL/libmpv 渲染 API，并保持项目对当前 devkitPro/libnx 工具链的兼容。

Architecture架构设计

The project is split into a protocol stack and a player stack. Protocol modules handle discovery, description, and control; player modules handle session state, playback backend, and rendering. 项目架构拆分为协议栈和播放器栈。协议模块负责发现、描述与控制；播放器模块负责会话状态、播放后端与渲染。 

main
  -> protocol/dlna
       -> discovery
       -> description
       -> control
  -> player
       -> core
       -> backend
       -> render

Playback Model播放模型

Instead of maintaining a separate pre-open model, the project keeps the playback path intentionally thin: control actions call the renderer, the renderer drives libmpv, and observed runtime properties are synced back into protocol state. 项目没有保留额外的预打开模型，而是采用更薄的播放路径：控制动作直接调用 renderer，由 renderer 驱动 libmpv，再将运行时属性同步回协议状态。 

SetAVTransportURI
  -> renderer_set_uri(...)
  -> libmpv loadfile
  -> observed runtime properties
  -> protocol_state sync

Key Technical Decisions关键技术决策

  • Kept protocol code standards-first and protocol-focused instead of mixing it with playback implementation details.坚持让协议代码以标准兼容性为优先，不把播放实现细节直接混入协议逻辑。
  • Used libmpv as the playback core so demuxing, decoding, and runtime state observation stay on the same engine.采用 libmpv 作为播放核心，使解复用、解码与运行时状态观测统一在同一引擎内完成。
  • Aligned protocol-facing state with renderer-observed runtime state to improve control fidelity across mixed DLNA control points.将协议可见状态与 renderer 观测到的运行时状态对齐，提升在不同 DLNA 控制端下的控制准确性。
  • Chose a direct renderer path and incremental refactoring strategy, separating structural refactors from behavior changes to keep the system stable.采用直接 renderer 路径与渐进式重构策略，把结构性重构与行为修改拆开推进，保障系统稳定性。
